\subsubsection*{\hspace{0.7cm}Transformación Lógica Espacial (\texttt{db-seeder.ts})}

Para poder incorporar los datos limpios en la base de datos, fue necesario realizar un cálculo matemático de las propiedades del grafo. En esta fase se utilizó la fórmula de Haversine (\texttt{calcularDistancia}) para determinar la distancia precisa en metros entre dos nodos que están conectados.

\begin{lstlisting}
function calcularDistancia(lat1: number, lon1: number, lat2: number, lon2: number): number {
  const R = 6371e3;
  const dLat = (lat2 - lat1) * Math.PI / 180;
  const dLon = (lon2 - lon1) * Math.PI / 180;
  const a = Math.sin(dLat/2) * Math.sin(dLat/2) +
            Math.cos(lat1 * Math.PI / 180) * Math.cos(lat2 * Math.PI / 180) *
            Math.sin(dLon/2) * Math.sin(dLon/2);
  const c = 2 * Math.atan2(Math.sqrt(a), Math.sqrt(1-a));
  return R * c;
}   
\end{lstlisting}

Además, al examinar la etiqueta de dirección (\texttt{oneway}), el script crea aristas bidireccionales o unidireccionales, repitiendo el segmento para posibilitar el cálculo matemático en las ambas direcciones. Por último, se concatenaron las coordenadas obtenidas en formato (\texttt{LINESTRING(lon lat)}) para que posteriormente puedan convertirse en objetos espaciales dentro de SQL Server.

\subsubsection*{\hspace{0.7cm}Carga Masiva a Azure SQL (\texttt{db-seeder.ts})}

Con el fin de prevenir errores por conexión y falta de memoria se estableció un procedimiento de reintentos (\texttt{executeWithRetry}) que se ocupa de registrar errores por tiempo de espera y que suspende la ejecución gradualmente antes de intentar nuevamente la inserción. Para la transferencia en red, los datos se cargaron al principio en una tabla de temporal (\texttt{ARISTAS\_STAGING}) utilizando operaciones \textit{Bulk Insert} en conjuntos de 50{,}000 registros. Después de que los datos se almacenaron de manera temporal en el servidor, se realizaron consultas iterativas con el fin de moverlos a la tabla principal (\texttt{ARISTAS}). Para convertir el texto en una geometría espacial indexable se utilizó la función \texttt{geography::STGeomFromText} y para cada tramo se asignaron los valores predeterminados del peso de riesgo.

Para el backend, estructurado modularmente con NestJS y TypeScript, el punto de entrada para las peticiones de navegación se construyó en la clase \texttt{ruteo.controller.ts}. \subsubsection{Implementación del Algoritmo A* Ponderado por Riesgo}

El servicio de aplicación es el lugar donde se concentra el núcleo del sistema, en el que se codificó el algoritmo A* para la búsqueda. La innovación de este componente consiste en cambiar la función de costo real $g(n)$, que se definió matemáticamente como la suma de la distancia de cada segmento multiplicada por su correspondiente peso de riesgo fórmula presentada en la Ecuación \ref{eq:gn}.

\begin{equation}
g(n) = \sum (distancia \cdot peso\_riesgo)
\label{eq:gn}
\end{equation}


Este peso parte de un valor base de 1.0 para secciones completamente seguras, aumenta en proporción $(N)$ a la peligrosidad del área.  La solución fue implementada en el servicio  \texttt{motor-algoritmo.service.ts}. 

\begin{lstlisting}
@Injectable()
export class MotorAlgoritmoService {
  private readonly logger = new Logger(MotorAlgoritmoService.name);
}  
\end{lstlisting}

Con el fin de lograr un funcionamiento eficaz del algoritmo A*, se creó una estructura de datos similar a una cola de prioridad, ubicada en el archivo \texttt{common/\allowbreak utils/\allowbreak min-heap.ts}, que garantiza tiempos de evaluación logarítmicos $O(log N)$ para determinar el camino más prometedor en cada iteración.

\begin{lstlisting}
export class MinHeap<T> {
  private data: T[] = [];

  constructor(private readonly comparar: (a: T, b: T) => number) {}

  get size(): number {
    return this.data.length;
  }
}
\end{lstlisting}


Al mismo tiempo, la función heurística $h(n)$, que orienta la búsqueda, delega el cálculo de distancias en superficies esféricas al módulo \texttt{haversine.ts}. El motor se comunica constantemente con grafo de adyacencia en memoria para consultar dinámicamente los grafos para la expansión de rutas mientras tanto \texttt{arista.repository.ts} utiliza previamente para construir ese subgrafo desde la base de datos (por Bounding Box).

\subsubsection{Penalización del Grafo Vial}
Una vez validada la estructura, el servicio invoca el procedimiento almacenado \texttt{dbo.SP\allowbreak\_Registrar\allowbreak\_Y\allowbreak\_Penalizar\allowbreak\_Incidente}. Para asegurar la atomicidad de la operación, la lógica se encapsuló dentro de un bloque \texttt{TRY...CATCH} con control explícito de transacciones (\texttt{BEGIN\allowbreak\ TRANSACTION\allowbreak\ /\allowbreak\ COMMIT}). A nivel espacial, el script transforma las coordenadas de latitud y longitud en un objeto nativo \texttt{geography::Point\allowbreak(lat,\allowbreak lon,\allowbreak 4326)}. Posteriormente, genera un polígono utilizando la función \texttt{.STBuffer\allowbreak(300)}.

\begin{lstlisting}
DECLARE @punto_incidente GEOGRAPHY = geography::Point(@latitud, @longitud, 4326);
INSERT INTO dbo.INCIDENTES (id_incidente, ubicacion, fecha_reporte, nivel_gravedad, tipo)
VALUES (@id_incidente, @punto_incidente, GETDATE(), @nivel_gravedad, @tipo);
DECLARE @area_busqueda GEOGRAPHY = @punto_incidente.STBuffer(300);
\end{lstlisting}

La penalización del grafo se ejecuta aplicando la función \texttt{.STIntersects()} para identificar qué vías cruzan el polígono de peligro, incrementando el campo \texttt{peso\_riesgo} de la tabla \texttt{ARISTAS} en una proporción de \texttt{nivel\_gravedad * 1000}. Para garantizar tiempos de respuesta en milisegundos sobre los millones de registros, se forzó la optimización de la ejecución inyectando la directiva \texttt{WITH(INDEX(\allowbreak SPATIAL\_IX\_Aristas\_Geometria))}.
\begin{lstlisting}
SET A.peso_riesgo = A.peso_riesgo + (@nivel_gravedad * 1000)
FROM dbo.ARISTAS A WITH(INDEX(SPATIAL_IX_Aristas_Geometria))
WHERE A.trayectoria.STIntersects(@area_busqueda) = 1;
\end{lstlisting}

\subsubsection{Limpieza de Incidentes Expirados}

Para prevenir que la red vial quede penalizada de forma permanente por reportes obsoletos, se construyó el servicio \texttt{IncidenteCleanupService}, utilizando una tarea programada esta orquesta el procedimiento \texttt{dbo.SP\_Limpiar\_Incidentes\_Expirados}, cuyo objetivo es identificar eventos con más de dos horas de antigüedad (\texttt{@horas\_caducidad = 2}).
\begin{lstlisting}
UPDATE a
SET a.peso_riesgo = 
CASE 
WHEN a.peso_riesgo - (i.nivel_gravedad
* 1000) < 1.0 THEN 1.0 
ELSE a.peso_riesgo - (i.nivel_gravedad * 1000) 
END
FROM dbo.ARISTAS a WITH(INDEX(SPATIAL_IX_Aristas_Geometria), ROWLOCK)
\end{lstlisting}
La reversión del riesgo se logró mediante una actualización condicional (CASE WHEN), la cual resta el factor de gravedad previamente sumado, asegurando matemáticamente que el valor de la vía jamás descienda del peso base de seguridad (1.0). Finalmente, los registros de la tabla INCIDENTES son purgado, esta actualización masiva empleó un bloqueo exclusivo a nivel de fila (ROW-LOCK), lo cual previene el bloqueo de la tabla completa y permite que el motor A* siga calculando rutas en paralelo. Dado que Android fuerza a cerrar aplicaciones que consumen recursos en segundo plano,  se orientó a gestionar adecuadamente los servicios de ubicación y a mitigar las imprecisiones del GPS para la traza de la ruta.
Asi pues este mismo identifica que el usuario nunca pierda la guía de la ruta segura y permite que el mapa muestre fluidamente el progreso recorrido.

\subsubsection{Seguimiento GPS Robusto en Segundo Plano}
Para garantizar el seguimiento continuo del usuario, se usa \texttt{TrackingService.kt} el cual es un servicio en primer plano, este se utiliza debido a que previene que el sistema operativo destruya el proceso cuando la aplicación se minimiza. 
\begin{lstlisting}
  class TrackingService : Service() {

    private lateinit var fusedLocationClient: FusedLocationProviderClient
    private val CHANNEL_ID = "TrackingServiceChannel"

    // Callback que se dispara cada vez que el GPS detecta movimiento
    private val locationCallback = object : LocationCallback() {
        override fun onLocationResult(locationResult: LocationResult) {
            super.onLocationResult(locationResult)
            locationResult.lastLocation?.let { location ->
                Log.d("TrackingService", "Ubicación actualizada: ${location.latitude}, ${location.longitude}")
                // Enviamos la ubicación al Singleton para que la UI reaccione
                LocationTracker.updateLocation(location)
            }
        }
    }
\end{lstlisting}

Paralelamente, se construyó el componente \texttt{LocationTracker.kt}, el cual opera como una instancia única encargada de encapsular el proveedor de ubicación fusionada (\texttt{FusedLocationProviderClient}) de Google Play Services. Este componente emite de forma continua y reactiva las coordenadas GPS a través de flujos asíncronos, permitiendo que el MapViewModel procese la información independientemente del estado de la vista.

\begin{lstlisting}
  object LocationTracker {
    private val _currentLocation = MutableStateFlow<Location?>(null)
    val currentLocation: StateFlow<Location?> = _currentLocation.asStateFlow()

    fun updateLocation(location: Location) {
        _currentLocation.value = location
    }
}
\end{lstlisting}


\subsubsection{Detección de Desvíos}
Con la libreria Google Maps Utility Library se introdujo la lógica para la navegacion áutonoma. Se empleó la función matemática \texttt{PolyUtil.isLocationOnPath} para evaluar constantemente si la coordenada actual del GPS intersectaba con la geometría de la ruta decodificada, aplicando una tolerancia de desvío estricta (entre 50 y 100 metros).

Para dotar al algoritmo de resiliencia frente a los "saltos" de antena satelital, se introdujeron lógicas de control críticas:
\begin{itemize}
  \item \textbf{Filtro de Precisión (Accuracy Filter):} Se configuró el sistema para ignorar silenciosamente las lecturas de GPS cuya dispersión de error fuera muy alta, previniendo falsos positivos de desvío.
  \item \textbf{Periodo de Gracia Inicial}: Se programó un bloqueador (Arming logic) que retrasa el monitoreo estricto durante los primeros segundos del viaje, permitiendo que el hardware estabilice su precisión inicial.
\end{itemize}

Al momento de detectar un desvío válido, el MapViewModel cambia al estado de Recalculating, en la cual solicita una nueva ruta segura al backend de forma transparente. Para el manejo de la transición visual no solo implica eliminar y volver a añadir la polilínea \texttt{map.clear()},  ya que, genera un parpadeo molesto. Para solucionar esto, se implementó una actualización de ``transición suave'' mediante el método \texttt{setPoints} sobre la instancia existente de la polilínea, logrando que el cambio de trayectoria sea instantáneo y fluido para el usuario.

El principal obstáculo fue la imprecisión del GPS en zonas de sombra estructural de la ciudad, lo que inicialmente disparaba recálculos innecesarios. dando como resultado que la implementación del filtro de precisión y el periodo de gracia logro estabilizar el comportamiento del software considerado este como el acierto más crítico del sprint, ya que estabilizó el consumo de red y mejoró drásticamente la experiencia del usuario.
